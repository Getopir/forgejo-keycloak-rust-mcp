# Configuration

Configure the daemon with these values:

- Keycloak issuer.
- OIDC discovery URL.
- MCP audience.
- Public MCP resource URL.
- Optional HTTPS guard: `--tls`, `--ssl`, or `FORGEJO_MCPD_TLS=true`.
- Bounded PR review output: `FORGEJO_MCPD_MAX_DIFF_BYTES` caps `get_pull_request_diff` text at 65536 bytes by default.
- Listen address.
- Forgejo base URL for Forgejo-backed tools.
- Forgejo connect and total request timeouts, defaulting to 5 and 30 seconds.
- Principal map path.
- Maximum page limit for bounded list responses.
- Approval store path for short-lived high-risk operation approvals.
- Approval TTL in seconds.
- Durable audit JSONL path through `FORGEJO_MCPD_AUDIT_LOG` or `--audit-log`.
- Per-agent token-bucket capacity, refill window, and maximum tracked-agent count.

Use environment variables or a deployment secret file outside the repository. Never commit tokens, client secrets, or private keys.

Use canonical URLs that operators, agents, OPIR-P, Keycloak, and the gateway can all resolve consistently. A container-only issuer such as `http://keycloak:8080/realms/master` is suitable only when every caller runs in the same DNS context. Prefer a stable LAN DNS name or HTTPS URL and reuse it in Keycloak issuer metadata, MCP protected-resource metadata, OPIR-P work-order inputs, and agent setup docs.

When Forgejo or the MCP public route is served over HTTPS, set the public MCP resource URL and Forgejo base URL to `https://` and enable the HTTPS guard. The guard validates URL configuration; TLS termination still belongs in the reverse proxy or TLS layer in front of the local daemon bind.

The principal map binds Keycloak `(issuer, subject)` to a Forgejo login and names the environment variable that contains that mapped user's Forgejo token. The map must not contain token values. Duplicate mappings and malformed token environment names are rejected at startup.

Agents should receive short-lived Keycloak access tokens for the MCP audience from a trusted broker or control plane. Do not give agents long-lived Forgejo personal access tokens as their MCP login.

Configured trusted identity headers, such as `X-WEBAUTH-USER`, are generated by the gateway. Requests that supply those headers directly to `/mcp` are rejected as spoof attempts.

Set `FORGEJO_MCPD_APPROVAL_STORE` to enable persistent approval validation. The approval store is append-only JSONL and must live outside the repository, outside web roots, and under filesystem permissions limited to the gateway service account. Approval consumption is serialized with a process mutex and an OS lock on the adjacent `.lock` file so concurrent gateway processes cannot consume one approval twice.

Approval-backed execution requires a different mapped principal from the approving principal. Do not reuse the same mapped Forgejo login for both roles.

Outbound Forgejo calls use `FORGEJO_MCPD_FORGEJO_CONNECT_TIMEOUT_SECONDS=5` and `FORGEJO_MCPD_FORGEJO_REQUEST_TIMEOUT_SECONDS=30` by default. Both values must be positive. Confirm remote state before retrying a mutation whose response timed out.

Enabled mappings with `principal_type: agent` are limited by an in-memory token bucket keyed by immutable Keycloak `(issuer, subject)`. `FORGEJO_MCPD_AGENT_RATE_LIMIT_REQUESTS` defaults to `60`, `FORGEJO_MCPD_AGENT_RATE_LIMIT_WINDOW_SECONDS` defaults to `60`, and `FORGEJO_MCPD_AGENT_RATE_LIMIT_MAX_AGENTS` defaults to `10000`. All bounds must be positive. Exhausted buckets return `429` with `Retry-After`; new agents fail closed if the tracking bound is full and no idle bucket can be removed.

The bucket is local to one daemon and resets on restart. Continue to enforce unauthenticated, body-size, concurrency, timeout, aggregate, human, and multi-instance limits at the reverse proxy.

Set `FORGEJO_MCPD_AUDIT_LOG=/var/log/forgejo-mcpd/audit.jsonl` to export structured audit events to persistent storage. The daemon refuses to start if the file cannot be opened. Each event is appended as one JSON line, flushed, and synchronized to the filesystem. Restrict access to the service account and audit readers, and configure host-level rotation and retention. Runtime write failures are reported through tracing.

The OIDC discovery document and JWKS are loaded once at startup. There is no TTL or automatic refresh. Follow [JWKS Cache Limits And Key Rotation](JWKS-Cache-Limits-And-Key-Rotation.md) when rotating signing keys.
