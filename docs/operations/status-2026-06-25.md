## Phase 1 Hardening And Phase 2 Baseline Update

`0.6.0` hardens the Phase 1 baseline and starts Phase 2:

- rejects duplicate principal mappings after issuer normalization;
- rejects empty required mapping fields;
- restricts `api_token_env` names to ASCII letters, digits, and underscore;
- rejects caller-supplied trusted identity headers on `/mcp`;
- adds server-capped `limit` and page-token `cursor` handling;
- implements `list_repository_issues`;
- implements `create_issue_comment`;
- implements `list_pull_requests`;
- implements `list_pull_request_reviews`;
- implements `list_releases`;
- implements `list_notifications`;
- returns approval-required responses for high-risk operations without an approval ID.

Remaining after `0.6.0`:

- persistent approval store;
- release creation execution;
- pull-request merge execution;
- admin and destructive execution;
- generated Forgejo API coverage after endpoint classification.

## Approval Store, Merge, And Release Execution Update

`0.8.0` added persistent, file-backed approval records for high-risk operations:

- exact operation, target, state, and body-hash binding;
- approver Keycloak subject and mapped Forgejo login recording;
- expiry enforcement;
- validation failures for missing, revoked, expired, mismatched, or wrong-principal approvals.

`0.9.0` made `merge_pull_request` the first executable high-risk operation:

- dry-run merge preview without Forgejo mutation;
- single-use approval consumption before the Forgejo merge API call;
- approver/executor separation by Keycloak subject and mapped Forgejo login;
- CLI support through `forgejo-mcpctl merge-pull-request`.

`0.10.0` extends the same approval execution model to `create_release`:

- dry-run release preview without Forgejo mutation;
- typed release payload fields for `tag_name`, `target_commitish`, `name`, `body`, `draft`, `prerelease`, and `hide_archive_links`;
- single-use approval consumption before the Forgejo release API call;
- CLI support through `forgejo-mcpctl create-release`.
